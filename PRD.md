# Fundraising CRM — Product Requirements Document
## "Don't Lose Anyone"

### Problem

Theo is running a high-volume fundraise ($15M now, ~$50M later). 30-50 investors in various stages, multiple calls per day, email threads with each, follow-up commitments, DD questions, material requests. The failure mode is simple: someone falls through the cracks. A warm intro goes cold because no one followed up. A DD question sits unanswered. A partner who asked for the data room never got it.

The current system (Review V2) has a D1 database and basic action cards. But it's a CRUD app with no intelligence. No agent pipelines. No transcript processing. No draft emails. No daily briefings. No learning.

### Vision

A fundraising CRM where AI agents do the bookkeeping and Theo does the relationships. Every morning: a briefing of who needs attention today and why. Before every call: a one-page prep with full history. After every call: drop the transcript and the system extracts everything. Draft emails ready to review and send. Nothing falls through the cracks because the system is watching everything.

### Architecture

#### Stack
- **Frontend**: Static HTML/CSS/JS served from Cloudflare Pages. ES5 JavaScript. No frameworks. Behind CF Access.
- **Backend**: Cloudflare Workers (API + agent orchestration)
- **Database**: Cloudflare D1 (SQLite)
- **AI**: Anthropic Claude API called directly from Workers (NOT through OpenClaw)
- **Auth**: Cloudflare Access (existing)

#### Why NOT OpenClaw agents?
OpenClaw's agent spawning is unreliable for production pipelines:
- Sessions compact and lose context
- Sub-agents are ephemeral with no persistent learning
- Tool availability depends on gateway state
- No structured error handling or retry
- Config lives in markdown files, not a database

The CRM's agents are Cloudflare Workers that:
- Call Claude API directly with structured prompts
- Store their config and learning in D1
- Have deterministic retry and error handling
- Are editable from the dashboard UI
- Survive deploys, restarts, and bad days

#### Worker Constraints to Design Around
- **CPU time limit**: Workers have 30s CPU time. Long Claude calls must use streaming. For structured extraction (transcript ingestion), use Cloudflare Queues to decouple request from processing — enqueue the job, process async, notify when done.
- **No TCP sockets**: Workers can't do IMAP. Email sync runs on the Mac mini (himalaya cron) pushing to the CRM API, or uses Cloudflare Email Workers for inbound-only.
- **D1 write contention**: Serialize agent writes through a single queue consumer to avoid WAL contention under concurrent agent calls.
- **API key storage**: Anthropic API key stored as Worker secret (`wrangler secret put ANTHROPIC_API_KEY`). Never in source.
- **Rate limiting**: Max 3 concurrent Claude API calls per Worker. Queue additional requests. Prevents both rate limit errors and cost spikes.

## User Stories

### Daily Operations

#### US-1: Daily Briefing
**As** Theo, **I want** a daily briefing showing who needs attention today and why, **so that** no investor falls through the cracks.

**Acceptance criteria:**
- Morning briefing generated automatically (or on demand)
- Prioritized list: overdue follow-ups first, then scheduled follow-ups, then stale investors
- Each entry: investor name, stage, last contact date, why they need attention, suggested action
- Briefing style is configurable (stored in D1, editable from UI)
- Delivered to #fundraising-briefings Discord channel AND visible in dashboard
- Mobile-optimized (375px)

#### US-2: Superhuman Card Processing
**As** Theo, **I want** to rapidly triage investor cards every day, **so that** I can process 30+ investors in 5 minutes.

**Acceptance criteria:**
- Card queue sorted by priority (overdue → due today → upcoming)
- Each card shows: name, stage, last contact, days since contact, next action, key context (one line)
- Keyboard navigation: ←→ or j/k to move, Enter to open detail, s to snooze, d to mark done, e to open email draft
- Swipe on mobile: right = done, left = snooze
- Sub-second transitions between cards
- Card count badge: "3 of 47" with progress indicator
- Color coding: red (overdue), amber (due today), green (on track), grey (snoozed)

#### US-3: Call Prep
**As** Theo, **I want** a one-page briefing before any investor call, **so that** I walk in fully prepared.

**Acceptance criteria:**
- Triggered by: clicking "Prep" on an investor card, or asking in Discord "#prep Bessemer"
- Shows: fund overview, key people, thesis, last call summary, all open questions, DD items outstanding, materials sent, email thread summary
- Generated by Briefing Agent using full D1 context
- Briefing format is configurable
- Printable / shareable as PDF

#### US-4: Transcript Ingestion
**As** Theo, **I want** to drop a call transcript and have the system extract all relevant information, **so that** I don't have to manually log call outcomes.

**Acceptance criteria:**
- Upload transcript via: dashboard upload, Discord message, or API
- Transcript Agent extracts:
  - Action items (with who and deadline)
  - Key concerns raised by investor
  - Sentiment assessment (excited / neutral / cautious / skeptical)
  - Stage signals (should stage move?)
  - Questions asked that need answers
  - Follow-up commitments ("I'll send the model Thursday")
  - People mentioned (new contacts to add?)
- Extracted data stored as structured records in D1
- Creates interaction record with full transcript + extracted summary
- Creates action items automatically
- Flags stage-change signals for Theo's confirmation (conservative — never auto-move)

#### US-5: Draft Emails
**As** Theo, **I want** draft follow-up emails ready to review after every call, **so that** I can send them in one click.

**Acceptance criteria:**
- Email Agent generates draft based on: last interaction, action items, investor context, Theo's voice
- Drafts appear on the investor card's right panel
- One-click to copy to clipboard (for pasting into Gmail)
- Edit in-place before copying
- Voice/style configurable per email type:
  - Post-call follow-up
  - DD response
  - Deck/material send
  - Cold follow-up (re-engagement)
  - Introduction thank-you
- Agent learns from edits: if Theo consistently changes something, store the correction

#### US-6: Email Thread Context
**As** Theo, **I want** to see the email history with an investor alongside their card, **so that** I have full context.

**Acceptance criteria:**
- Email sync from fc@tycho.sh (himalaya or direct IMAP from Workers — TBD)
- Thread view showing sent/received emails with timestamps
- Emails linked to investor record
- Email Agent reads thread before drafting reply

### Pipeline Management

#### US-7: Pipeline Kanban
**As** Theo, **I want** a visual pipeline showing all investors by stage, **so that** I see the full picture.

**Acceptance criteria:**
- Columns: Lead → Intro → Meeting → DD → IC → Term Sheet → Closed
- Drag-and-drop to move between stages (with confirmation)
- Card preview on each: name, days in stage, next action
- Filter by: priority, sector focus, check size, intro source
- Stage velocity indicators (avg days in stage)
- Count and total check size per column

#### US-8: Stale Investor Detection
**As** Theo, **I want** automatic alerts when an investor hasn't been contacted within their expected cadence, **so that** no one goes cold.

**Acceptance criteria:**
- Configurable cadence per stage:
  - Intro: 3 days
  - Meeting: 5 days
  - DD: 3 days
  - IC: 2 days
  - Term Sheet: 1 day
- Stale investors surfaced in daily briefing AND as dashboard notification
- Escalation: after 2x cadence, flag as critical
- Snooze capability ("they're on holiday, check back in 10 days")

#### US-9: Follow-up Scheduling
**As** Theo, **I want** to record follow-up commitments with dates, **so that** the system reminds me.

**Acceptance criteria:**
- "Follow up on Thursday" creates a scheduled action
- Shows in daily briefing on the right day
- Can be created from: card UI, transcript extraction, Discord command
- Overdue follow-ups are the highest priority in card queue

#### US-10: Introduction Tracking
**As** Theo, **I want** to track who introduced me to whom, **so that** I can send thank-yous and ask for more intros.

**Acceptance criteria:**
- Intro source field on investor record (person or firm)
- Intro chain visible: "Alice at EF → introduced Bob at Bessemer"
- Thank-you reminder: 48h after intro, suggest thank-you note if not sent
- Intro effectiveness: which sources lead to further stages?

### Agent System

#### US-11: Agent Configuration UI
**As** Theo, **I want** to configure each agent's behavior from the dashboard, **so that** I can tune their output without editing code.

**Acceptance criteria:**
- Settings page with agent cards:
  - **Pipeline Agent**: stage definitions, cadence rules, priority weights
  - **Briefing Agent**: briefing template, emphasis areas, format preferences
  - **Email Agent**: voice profile, templates per email type, correction history
  - **Transcript Agent**: extraction categories, confidence thresholds, auto-action rules
- Each agent has:
  - System prompt (editable, with version history)
  - Example outputs (few-shot examples stored in D1)
  - Correction log (every time Theo edits agent output, the before/after is stored)
  - Toggle: active/paused

#### US-12: Agent Learning
**As** Theo, **I want** agents to learn from my corrections, **so that** they get better over time.

**Acceptance criteria:**
- When Theo edits a draft email → before/after stored as correction in D1
- When Theo changes a briefing → correction stored
- When Theo overrides a transcript extraction → correction stored
- Most relevant corrections (max 10) selected by recency + similarity and included as few-shot examples in future agent calls. Not all corrections — context window budget is 5k tokens for examples.
- Correction count visible per agent ("47 corrections, last: 2h ago")
- Can review and prune corrections from UI
- Corrections older than 90 days auto-archive unless pinned

### Data Management

#### US-13: Contact Management
**As** Theo, **I want** to track multiple contacts per fund with their roles and context.

**Acceptance criteria:**
- Multiple contacts per investor
- Fields: name, title, email, phone, role (decision maker / champion / analyst), notes
- Primary contact designation
- "Copy all emails" for fund (for group sends)
- Contact mentioned in transcript → auto-link or suggest

#### US-14: DD Question Tracking
**As** Theo, **I want** to track DD questions and their status.

**Acceptance criteria:**
- Questions linked to investor
- Status: asked → in progress → answered → shared
- Answer field (or link to material)
- Questions extracted from transcripts automatically
- Dashboard view: all open DD questions across all investors

#### US-15: Material Tracking
**As** Theo, **I want** to know what materials each investor has received.

**Acceptance criteria:**
- Track: which deck version, data room access, reference pages, financial model
- Auto-log when materials are sent via email
- "Has NOT seen latest deck" flag when deck is updated

### Integrations

#### US-16: Discord Integration
**As** Theo, **I want** to interact with the CRM from Discord.

**Acceptance criteria:**
- Commands in #fundraising-briefings:
  - "Briefing" → daily briefing
  - "Prep [investor]" → call prep
  - "Update [investor] to [stage]" → stage change
  - "Log: [investor] - [notes]" → create interaction
  - "Draft email [investor]" → generate and post draft
- Transcript drop: upload transcript file in channel (Discord file upload, not paste — transcripts exceed 4000 char message limit) → ingestion pipeline triggers
- Daily briefing auto-posted at configured time

#### US-17: Email Integration
**As** Theo, **I want** emails synced into the CRM.

**Acceptance criteria:**
- Sync from fc@tycho.sh inbox
- Match emails to investor records (by email address)
- Show in investor timeline
- Flag unread investor emails in daily briefing

## Database Schema (Evolution from Review V2)

### Existing tables (keep, extend):
- `investors` — add: intro_source, intro_person, snooze_until, stale_threshold_days
- `contacts` — add: role enum (decision_maker, champion, analyst, other)
- `interactions` — add: transcript_raw, transcript_extracted (JSON), sentiment, follow_up_date
- `actions` — add: source (manual, transcript, email, system), scheduled_date
- `materials` — add: deck_version, date_shared
- `dd_items` — add: extracted_from_interaction_id

### New tables:
- `agent_configs` — id, agent_type (pipeline|briefing|email|transcript), system_prompt, config_json, version, updated_at
- `agent_corrections` — id, agent_type, input_context, original_output, corrected_output, created_at
- `agent_examples` — id, agent_type, input, output, active, created_at
- `email_threads` — id, investor_id, subject, last_message_at, thread_json
- `emails` — id, thread_id, investor_id, direction (inbound|outbound), from_addr, to_addr, subject, body, date, message_id
- `follow_ups` — id, investor_id, description, due_date, status (pending|done|overdue), created_from (interaction_id or manual)
- `briefings` — id, type (daily|call_prep), investor_id (null for daily), content, generated_at
- `transcripts` — id, investor_id, interaction_id, raw_text, extracted_json, processed_at

## Agent Pipeline Architecture

### Pipeline 1: Transcript Ingestion
```
Input (raw transcript) 
  → Transcript Agent (Claude API)
  → Structured extraction (JSON)
  → Write to D1: interaction, action items, DD questions, follow-ups
  → If stage signal detected: flag for confirmation
  → If email draft needed: trigger Email Agent
```

### Pipeline 2: Daily Briefing
```
Cron trigger (8am London) OR manual request
  → Pipeline Agent reads: all investors, overdue actions, due follow-ups, stale contacts
  → Briefing Agent formats using template from D1
  → Post to Discord #fundraising-briefings
  → Store in D1 briefings table
```

### Pipeline 3: Call Prep
```
Trigger: UI button or Discord "#prep Bessemer"
  → Pipeline Agent gathers: all interactions, emails, DD items, materials, contacts
  → Briefing Agent formats as call prep using template
  → Return to UI or post to Discord
```

### Pipeline 4: Email Draft
```
Trigger: UI button, post-transcript, or Discord "#draft Bessemer"
  → Email Agent reads: investor context, last interaction, action items, email thread
  → Loads corrections/examples from D1
  → Generates draft in Theo's voice
  → Returns to UI right panel or Discord
```

### Pipeline 5: Stale Detection
```
Cron trigger (daily)
  → Pipeline Agent: check all active investors against stage cadence
  → Flag stale, escalate critical
  → Include in daily briefing
  → Update action cards
```

## Non-Functional Requirements

- **Performance**: card transitions < 100ms, page load < 2s. Agent calls: email drafts < 15s (streaming), transcript extraction < 30s (async with progress indicator), call prep < 20s (streaming). All agent calls show streaming output, never a blank wait.
- **Auth**: CF Access on all routes, no public pages.dev access
- **Data**: All state in D1, survives deploys
- **Reliability**: Agent failures are non-fatal, surface error to UI, retry available
- **Mobile**: All views work at 375px. Daily briefing and card processing are phone-primary.
- **Style**: Light theme, system fonts, Müller-Brockmann grid, type scale 42/22/16/14/13/12/11px
- **Code**: ES5 JavaScript frontend, no frameworks, no build step
- **Deploy**: via scripts/deploy-pages.sh (Pages) + wrangler for Workers
